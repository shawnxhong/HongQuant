HongQuant compact dev conversation summary
Date: 2026-04-13

Purpose
- Personal mid/low-frequency quant system.
- Initial target coverage: US equities/ETFs, crypto, and later A-share/ETF support.
- Execution style: semi-automatic. System generates signals and recommendations; user decides how to place trades.
- Constraint: single-operator project, so infrastructure should stay lightweight and reuse mature open-source tools.

Core decisions from the original conversation
- Prioritize a maintainable Python stack over heavy institutional infrastructure.
- Build data and research foundations first; do not start with full automated execution.
- Keep a unified abstraction across markets where possible; split only at adapters and execution endpoints.
- Run quantitative price/volume research and LLM-assisted research in parallel, then merge them later at portfolio decision time.
- Use TradingView as a charting/prototyping/alert source, not as the primary data source or final backtesting engine.

Recommended architecture agreed in the conversation
- Data layer: market data, macro data, filings, alternative inputs.
- Research layer:
  - B1 quantitative strategies, factors, and backtests
  - B2 LLM earnings/filings valuation and recommendation pipeline
- Decision layer: risk control, portfolio construction, target weights.
- Delivery layer: daily reports, notifications, position suggestions.

Recommended infrastructure choices
- Language/tooling: Python + uv + ruff + pytest.
- Data storage: Parquet + DuckDB for historical research data.
- Metadata/ops storage: Postgres optional.
- Orchestration: Prefect.
- Notifications: Telegram or Feishu webhook.
- LLM memory/retrieval: LanceDB.
- Dashboard later: Streamlit.

Recommended reusable open-source components
- US market data: Alpaca, yfinance, SEC EDGAR.
- Crypto market data: ccxt.
- Macro data: FRED.
- Research/backtest: vectorbt first, Backtrader optional, Qlib as a possible research layer.
- Portfolio optimization: PyPortfolioOpt.
- LLM pipeline: LangChain or similar orchestration around filing ingestion and valuation logic.

TradingView conclusion
- Use it for charting, Pine prototypes, and alerts to webhook.
- Do not use it as the system's core data source.
- Do not rely on it as the canonical backtest engine.

Planned phased roadmap from the conversation
- Phase 0: scaffold package, data lake, adapters, flows, tests, CI.
- Phase 1: first quantitative signal loop, likely CTA/cross-sectional momentum plus risk controls and target weights.
- Phase 2: LLM filing/valuation pipeline and recommendation synthesis.
- Phase 3: TradingView webhook integration, dashboard, and possible execution upgrades.

Notable implementation details
- The data lake writes OHLCV bars partitioned by market, asset type, interval, and year.
- Writes are idempotent by `(symbol, ts)`.
- Daily equities flow uses Alpaca as primary source and yfinance as fallback/backfill option.
- Hourly crypto flow uses ccxt.
- FRED and EDGAR adapters exist as research inputs but are not yet wired into full downstream pipelines.
- Many package namespaces such as `backtest`, `factors`, `strategies`, `portfolio`, `llm`, and `report` are placeholders for future phases.
